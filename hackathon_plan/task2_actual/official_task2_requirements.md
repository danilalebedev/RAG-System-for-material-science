# Требования из актуального условия Task 2

Источник: https://nornickel-ai-hackathon.ru/task-2

## Суть задачи

Нужно разработать поисково-аналитическую систему для анализа научно-технической информации. Система должна связывать статьи, эксперименты, материалы, свойства, режимы, установки, исследовательские команды и выводы.

Ключевой пользовательский вопрос: что уже делали по объекту/материалу/условию X, каким был режим Y, какой эффект наблюдался на свойство/KPI Z, какие есть связанные сущности, пробелы, противоречия и история решений.

## Новые жесткие требования

| Требование | Следствие для архитектуры |
|---|---|
| Сложные запросы должны отрабатывать за 3-5 секунд при объеме до 1 млн сущностей | Нужны предрассчитанные индексы, не agentic full-scan; online LLM только на коротком evidence pack |
| Поддержка числовых диапазонов: температура, давление, концентрация, коэффициенты | Нужны typed fields и numeric indexes, а не только vector search |
| География: месторождения, регионы, страны, координаты/гео-привязка | Нужны geo entities, geo aliases, возможно bounding/region filters |
| История и версии фактов | Нужны `valid_from`, `valid_to`, `source_date`, `version`, conflict handling |
| Связи между сущностями и поиск цепочек | Нужен property graph/hypergraph вокруг эксперимента и процесса |
| Достоверность, источники, citation/provenance | Every fact/edge must have evidence chunk/table/page/source |
| Роли доступа, безопасность, аудит | Нужны RBAC-модель, audit log, masking/export controls хотя бы в проектной архитектуре |
| Экспорт результатов | Нужны CSV/JSON/GraphML/PNG/PDF report endpoints |
| Корпус содержит научные документы разных типов | Ingestion PDF/DOCX/PPTX/XLS/архивов обязателен |

## Deliverables, которые надо показать

1. Рабочий прототип или воспроизводимый pipeline.
2. Индексация предоставленного корпуса или демонстрационного подмножества.
3. Интерфейс для аналитического запроса.
4. Граф связанных сущностей.
5. Ответы с источниками.
6. Поиск пробелов и противоречий.
7. Метрики качества и скорости.
8. План масштабирования до 1 млн сущностей.
9. Учет ограничений по моделям/API и ресурсам.

## Критичные проектные решения

1. Не делать только "чат по PDF". Он не покроет числовые диапазоны, графовые цепочки, версии и RBAC.
2. Не делать только graph DB. Без full-text/vector слоя плохо работает поиск по длинным документам.
3. Делать гибрид:
   - full-text index;
   - vector index;
   - typed metadata store;
   - graph store;
   - LLM только для extraction, summarization и ответа по evidence.
4. Разделить offline и online:
   - offline: parsing, extraction, graph build, indexes;
   - online: query parse, index lookup, graph expansion, rerank, short answer generation.
